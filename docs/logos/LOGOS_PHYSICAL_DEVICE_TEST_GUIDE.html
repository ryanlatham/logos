Privacy and safety

Before you start

Workspace

/Users/ryan/Development/logos

Final report: docs/logos/FINAL_REPORT.md

Device

Physical iPhone, USB or Wi-Fi debugging, Apple Developer signing team, and Tailscale/private LAN reachability to the Mac.

Required secret

LOGOS_DEVICE_SECRET must match between the adapter and app. In Logos, use Settings → Device key → Edit to paste an adapter-provided secret, or Generate a phone key and restart the adapter with that same value locally. Never paste the actual value into the report.

Optional APNS

LOGOS_APNS_* credentials are needed only for live Apple push delivery. The app can still test foreground flows without them.

Recommended order

First run the mock adapter over the private address to prove phone ↔ Mac transport and UI. Then enable the real Hermes plugin and repeat the core flows. If the mock path fails, do not waste time debugging Hermes yet; the network/app path is guilty.

Setup commands

Find a private Mac address
tailscale ip -4 2>/dev/null || ipconfig getifaddr en0

Use this address in the iPhone app as ws://<private-mac-ip>:8765.

Start the mock adapter for hardware transport/UI smoke test
cd /Users/ryan/Development/logos
export LOGOS_DEVICE_SECRET='[REDACTED]'
PYTHONPATH=/Users/ryan/Development/logos/plugins:/Users/ryan/.hermes/hermes-agent \

cd /Users/ryan/Development/logos/clients/ios/Logos

Manual test scenarios

Run these in order. Mark each status as you go; the report form will include the scenario matrix.

Scenario 1

Build, sign, and launch on iPhone

Open Logos.xcodeproj and set signing for the Logos target.Expected: Xcode accepts the development team and the physical iPhone is a valid run destination.
Build and run Logos on the phone.Expected: the app launches to the Logos screen with nav bar, Settings gear, project picker, message thread, and composer/voice controls.
If iOS asks to trust the developer or device, complete the normal Apple trust flow.Expected: no crash on first launch; no notification, microphone, or speech prompt should be forced until you use those features.
Scenario 2

Connect over Tailscale/private LAN

Start the mock adapter bound to the Mac private address.Expected: terminal prints READY ws://... with secret=[REDACTED].
In Logos Settings, set Adapter to ws://<mac-tailnet-or-private-lan-ip>:8765, then use Device key → Edit to paste the matching shared secret.Expected: the key row shows only a masked suffix after Save.
Tap Connect.Expected: status changes from Connecting to Connected. If iOS asks for local-network access, allow it for this private-network test.
Tap Disconnect, then Reconnect.Expected: reconnect succeeds without app relaunch and without duplicate project/message spam.
Scenario 3

Typed text, project picker, and slash-command pass-through

Type phone smoke test and tap Send.Expected: user bubble appears, then assistant response renders. Mock adapter response should say Mock Hermes received: phone smoke test.
Create a project named phone-smoke using New project title + New.Expected: project picker can switch to the new project and message history stays scoped.
With the real Hermes plugin later, send a benign slash command such as /status.Expected: Logos does not swallow the slash command; Hermes responds through the normal gateway path.
With a known titled desktop session, test /resume <title-or-id>.Expected: Hermes resumes or reports a clear resolution issue. No separate Logos-only resume behavior should appear.
Scenario 4

Approval and clarification cards

Using the mock adapter, send /mock_approval.Expected: an approval card appears with Approve and Deny. The fixture says no command is executed.
Tap Deny, then repeat and tap Approve.Expected: the card clears each time; app remains connected.
Send /mock_clarify.Expected: a clarification card appears with choices and an Answer field.
Answer using both a choice button and free text on repeat.Expected: the card clears and no duplicate answer appears.
Repeat one real Hermes task that naturally asks for approval/clarification if available.Expected: the phone card maps back to Hermes semantics. No hidden auto-approval.
Scenario 5

TTS / summary playback

Generate an assistant message, then tap Play under it.Expected: playback status changes to Requesting audio, then Receiving audio, then Playing audio.
Listen through phone speaker and, if convenient, Bluetooth output.Expected: mock adapter WAV audio plays on the transport path; real Hermes plugin playback should be intelligible macOS say speech. Simulator playback remains protocol/status validation only.
Tap Play repeatedly on the same message.Expected: no crash, no permanent stuck Receiving audio state.
Scenario 6

Hold-to-talk speech recognition

Press and hold Hold to Talk.Expected: iOS requests Microphone and Speech Recognition permission if not already granted; Voice status changes to listening/recording.
Say: what is the current project?, then release.Expected: partial transcript appears while speaking; exactly one final speech message is sent on release.
Hold with silence, then release.Expected: no empty final turn reaches Hermes.
Deny permissions if you can safely reset them for a second pass.Expected: app disables voice or shows a clear error; it does not silently fall back to network recognition.
Scenario 7

Tap-to-talk silence detection and edge cases

Tap Tap to Talk, speak a short request, then stop speaking.Expected: energy/silence detection stops recording and sends one final transcript.
Tap Tap to Talk and stay silent.Expected: app stops after initial silence timeout and sends no empty final turn.
Tap Tap to Talk, speak, then tap Stop Tap manually.Expected: final transcript dispatches once.
Rapidly tap voice controls several times.Expected: only one recording session is active; app does not crash or wedge.
Scenario 8

Run status, queue behavior, and stop/cancel

With the real Hermes plugin, start a task that runs long enough to observe status.Expected: status leaves Idle and returns to Idle on completion.
While running, tap Stop.Expected: run_cancel maps to Hermes /stop; UI reaches cancelling/idle or shows a clear failure.
Send a same-project follow-up while a run is active.Expected: Hermes queue/guard semantics apply; Logos should not persist fake completed messages.
Scenario 9

Private APNS notification and reconnect route

Start adapter/Hermes with APNS credentials set and the Xcode signing team configured for Push Notifications.Expected: the app target uses Logos/Logos.entitlements with aps-environment=development; APNS client initializes for sandbox. Missing credentials should skip live sends without breaking normal responses.
Tap Enable in the Notifications panel and allow the iOS prompt.Expected: app receives a device token and sends register_device; normal logs do not print raw token.
Background the app and trigger an assistant completion.Expected: notification title Hermes finished, body Open Logos to view the result.
Tap the notification.Expected: Logos opens to the relevant project and performs reconnect + delta sync. Payload contains routing ids only, not response text or command previews.
Repeat for approval and clarification if practical.Expected: private alert titles Hermes needs approval / Hermes needs clarification, then card fetch after reconnect.
Scenario 10

Background reconnect, force-close, and log privacy

Send a message, background the app for at least 30 seconds, then reopen.Expected: app reconnects or shows a clear connection state; missed messages sync without duplicates.
Force-close and relaunch Logos.Expected: local message store shows recent messages; reconnect + messages_get refreshes state.
Inspect adapter/Hermes logs after the run.Expected: no raw APNS tokens, no shared secret, no API keys, no unrelated private transcript dumps in normal logs.

Troubleshooting

SymptomLikely causeCheck
Phone never connectsWrong bind address, firewall, Tailscale/LAN path, or mismatched secret.Confirm adapter printed READY on the same IP/port; verify iPhone can reach Mac; retype secret locally without reporting it.
Connected but Send does nothingWebSocket authenticated but frame rejected or adapter crashed.Watch adapter terminal for errors; confirm app still shows Connected; retry with mock adapter.
Voice disabledOn-device speech recognition unavailable or permissions denied.This is acceptable if clearly surfaced. It is a bug only if the app silently falls back to network recognition.
No APNS notificationCredentials, signing team capability, device token registration, sandbox/prod mismatch, or app/background state.Confirm Logos/Logos.entitlements is attached, record APNS environment and bundle id, but never paste tokens or key material.
Duplicate messages after reconnectDelta sync/server_seq/message_id dedupe bug.Record exact sequence: background duration, project, whether force-close occurred, screenshot if sanitized.
Playback stuck Receiving audioMissing audio_end, chunk decode failure, or audio session issue.Try another assistant message; note whether status changes to Playing audio.
